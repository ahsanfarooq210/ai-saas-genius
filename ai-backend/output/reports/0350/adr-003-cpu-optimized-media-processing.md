# ADR-003: CPU-Optimized Media Processing Pipeline

## Status
Accepted

## Context
The social media automation platform must ingest user-generated photos and videos, normalize them to platform-specific formats, and deliver them to external APIs. Video transcoding (resolution scaling, codec normalization, thumbnail extraction) is CPU-bound and memory-intensive. Running FFmpeg inside the Node.js/Express service or generic job workers would saturate the event loop, exhaust memory, and prevent independent scaling of the API tier. We need a processing topology that isolates CPU-heavy work, preserves upload responsiveness, and scales cost-effectively with queue depth.

## Decision
We will implement a dedicated, asynchronous media processing pipeline decoupled from the request-serving path:

- **media_service** acts as an upload proxy and metadata tracker. It accepts multipart uploads, validates MIME types and size limits, streams originals to **object_storage**, records asset metadata in **mongodb_ops**, and enqueues processing jobs to **redis_streams_queue**.
- **media_processor** runs as a standalone fleet of CPU-optimized workers. Each worker consumes from **redis_streams_queue**, downloads the original blob from **object_storage**, executes FFmpeg pipelines on local ephemeral storage, uploads processed variants back to **object_storage**, updates job state in **mongodb_ops**, and ACKs the stream entry.
- **object_storage** stores immutable originals and derived variants under distinct key prefixes with lifecycle policies.
- **cdn** serves processed media via edge-cached URLs and optimizes presigned delivery to end users and platform APIs.
- **job_worker** does not perform transcoding. It consumes downstream publish jobs that reference the completed output keys produced by **media_processor**.

## Responsibilities

- **media_service**
  - Validate upload headers, virus scan stubs, and platform-specific format allow-lists.
  - Generate time-limited presigned PUT URLs for direct-to-storage client uploads.
  - On upload completion, create a `MediaAsset` record in **mongodb_ops** with status `pending_processing`.
  - Serialize a processing job and XADD to `stream:media:processing`.
  - Serve presigned GET URLs and CDN URLs for processed assets to the client and **publisher_service**.

- **media_processor**
  - Maintain a persistent consumer group `ffmpeg-workers` on `stream:media:processing`.
  - Download source blobs to local ephemeral disk.
  - Execute FFmpeg commands based on job parameters (e.g., H.264 to MP4, 1080p/720p variants, 30 fps cap, stereo AAC audio).
  - Upload output blobs to **object_storage** under the `processed/` prefix.
  - Update the corresponding `MediaJob` document in **mongodb_ops** to `completed` and write a transient status key to **redis_cache** for polling clients.
  - Clean local scratch files immediately after upload to prevent disk exhaustion.

- **object_storage**
  - Durably store original blobs with object-lock/immutability.
  - Store processed variants with standard redundancy.
  - Support multipart upload and ranged reads for resumable operations.

- **redis_streams_queue**
  - Provide an ordered, append-only job log for media tasks.
  - Support consumer groups so multiple **media_processor** instances can claim entries without race conditions.
  - Retain acknowledged entries long enough for audit (default 24 hours).

- **job_worker**
  - Consume publish tasks that depend on `outputKeys` generated by **media_processor**.
  - Enforce per-user concurrency limits so a single user cannot flood the transcoding fleet.

## APIs / Interfaces

- **Upload Initiation**
  - `POST /media/upload` → **media_service** returns `{ uploadUrl, assetId, fields }` (presigned POST or PUT).
  - Client uploads directly to **object_storage**.

- **Processing Trigger**
  - **media_service** executes:
    ```redis
    XADD stream:media:processing * \
      jobId <uuid> \
      userId <uuid> \
      sourceKey "originals/{userId}/{assetId}/{filename}" \
      targetFormats "[1080p_mp4,720p_mp4,thumbnail_jpg]" \
      metadata "{...}"
    ```

- **Worker Consumption**
  - **media_processor** blocks on:
    ```redis
    XREADGROUP GROUP ffmpeg-workers {instanceId} COUNT 1 BLOCK 5000 STREAMS stream:media:processing >
    ```
  - On success, ACKs with `XACK stream:media:processing ffmpeg-workers {entryId}`.
  - On failure after 3 deliveries, moves entry to `stream:media:deadletter` via `XADD`.

- **Status Polling / Cache**
  - **media_processor** writes:
    ```redis
    SET media:status:{jobId} "{\"status\":\"completed\",\"outputKeys\":[...]}" EX 3600
    ```
  - Clients poll `GET /media/status/{jobId}`, which reads from **redis_cache** before falling back to **mongodb_ops**.

- **Delivery Interface**
  - `GET /media/{assetId}/url?variant=1080p_mp4` → **media_service** returns a CDN URL or short-lived presigned GET URL from **object_storage**.

## Data Owned

- **Original Media Blobs**
  - Path: `originals/{userId}/{assetId}/{filename}`
  - Immutable; retained for the life of the user account for re-transcoding.

- **Processed Variants**
  - Path: `processed/{assetId}/{variant}/{filename}`
  - Examples: `processed/{assetId}/1080p_mp4/video.mp4`, `processed/{assetId}/thumbnail_jpg/poster.jpg`.

- **Job State Documents (MongoDB)**
  - Collection: `mediaJobs`
  - Fields: `jobId`, `userId`, `sourceKey`, `outputKeys[]`, `ffmpegCommand`, `exitCode`, `stderrLog`, `status` (`pending` | `processing` | `completed` | `failed`), `retryCount`, `createdAt`, `completedAt`.

- **Asset Metadata Documents (MongoDB)**
  - Collection: `mediaAssets`
  - Fields: `assetId`, `userId`, `filename`, `contentType`, `sizeBytes`, `originalKey`, `variants[]`, `platformRestrictions`.

- **Ephemeral Scratch Data**
  - Local to **media_processor** instances: `/scratch/{jobId}/` containing source download, intermediate frames, and output files. Explicitly wiped after each job.

## Failure Modes

- **Transcoding Failure**
  - FFmpeg returns non-zero exit (corrupt source, unsupported codec, invalid parameters). Worker captures `stderr`, writes it to `mediaJobs.stderrLog`, sets status to `failed`, and does not ACK the stream entry until the retry count exceeds 3. After max retries, the entry is moved to `stream:media:deadletter` and an alert is fired.

- **Disk Pressure on CPU Instances**
  - Large 4K source files or long videos can exhaust ephemeral storage. Worker monitors `df` before claiming a job. If local disk >85%, it pauses `XREADGROUP`, finishes the current job, and signals the auto-scaler to drain and replace the instance with a larger ephemeral volume.

- **Object Storage Timeout**
  - Upload/download operations use exponential backoff with full jitter (base 1s, cap 60s, max 5 attempts). If **object_storage** is persistently unavailable, the worker releases the stream claim (negative ACK with `XCLAIM` idle time reset) so another worker can retry.

- **Queue Backpressure**
  - If `stream:media:processing` pending entry count exceeds 10,000, the monitoring system triggers an alert and scales out the **media_processor** fleet. **media_service** may temporarily reject new uploads with HTTP `429` and a `Retry-After` header to protect the queue.

- **Poison Pill**
  - A malformed message (missing `sourceKey`) causes the worker to crash. Redis Streams tracks delivery count via `XPENDING`. After 3 deliveries without ACK, an automated sweeper moves the entry to the dead-letter stream and notifies operations.

- **CDN Stale Content**
  - Re-processing an existing asset generates new variant keys under a content-addressed or versioned path, so cache collisions are rare. If an in-place update occurs, **media_service** issues cache invalidation by CDN tag or surrogate key.

## Scaling Considerations

- **CPU-Optimized Instance Fleet**
  - **media_processor** runs on dedicated compute-optimized instances (e.g., AWS c6i/c7g, GCP c2, Azure Fsv2) with a minimum 4 vCPUs per worker. Burstable or shared-core instance types are prohibited because sustained all-core load from FFmpeg causes credit exhaustion and throttling.

- **Horizontal Scaling Metric**
  - Auto-scaling is driven by the pending message count in the `ffmpeg-workers` consumer group. Target ratio: 1 worker per 500 pending messages. Scale-out cooldown: 60 seconds. Scale-in cooldown: 300 seconds to avoid thrashing.

- **Memory and Thread Tuning**
  - Each FFmpeg job is capped to 2 concurrent threads (`-threads 2`) to prevent a single job from monopolizing a worker. Workers run up to `floor(vCPUs / 2)` parallel jobs via worker pools or separate OS processes, ensuring predictable latency.

- **Network Throughput**
  - Workers require high-throughput networking (minimum 10 Gbps) to download multi-gigabyte source files and upload variants without becoming I/O bound. Instance placement is restricted to the same region as **object_storage** to minimize egress latency.

- **Ephemeral Storage vs. Block Storage**
  - Scratch space uses locally attached NVMe/ephemeral disks rather than network-attached block storage. This avoids I/O contention with the OS disk and reduces cost for temporary data. Originals and outputs are never stored locally longer than the duration of one job.

- **Per-User Concurrency Limits**
  - A single user cannot monopolize the fleet. **media_processor** checks a Redis-backed semaphore `media:concurrency:{userId}` (default limit: 2 concurrent jobs per user). If the limit is reached, the worker skips that entry and processes another user's job, ensuring fair scheduling.

- **Cost and Lifecycle**
  - Processed variants are accessed frequently for 7 days after creation, then transitioned to infrequent-access storage if not linked to a pending publish job. Originals remain in standard storage. CDN cache TTL for processed media: 1 year with immutable query parameters.

- **Warm Pools**
  - During predictable peak windows (e.g., early morning before scheduled posts), the fleet maintains a warm pool of 2 idle CPU-optimized instances to absorb burst uploads without cold-start latency.
