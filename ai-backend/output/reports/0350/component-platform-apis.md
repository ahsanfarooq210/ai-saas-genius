## component-platform-apis

### Responsibilities

`platform_apis` is the external egress boundary between the automation platform and third-party social media networks. It is not an internally deployed service but the aggregated surface area of external provider APIs that the system consumes. Its responsibilities include:

- **Request Translation**: Converting internal normalized publish payloads into platform-specific API formats (e.g., Meta Graph API multipart forms, LinkedIn URN-based JSON, Twitter/X v2 JSON).
- **Credential Injection**: Accepting OAuth tokens retrieved from the internal `token_vault` and injecting them into outbound requests as bearer tokens, query parameters, or signed headers per platform requirement.
- **Rate Limit Compliance**: Operating within platform-enforced quotas. It does not enforce limits itself, but it emits standardized rate-limit headers and error codes so the internal `rate_limiter` and `publisher_service` can throttle accordingly.
- **Retry & Idempotency Surface**: Exposing platform-specific retry semantics (exponential back-off windows, `Retry-After` headers) and honoring idempotency keys passed from the `publisher_service`.
- **Response Normalization**: Returning a uniform result structure to the `publisher_service` on success or failure, abstracting platform-specific error schemas.

### APIs and Interfaces

Because `platform_apis` represents external infrastructure, the "interface" documented here is the internal adapter contract used by the `publisher_service`, followed by the key external surfaces.

#### Internal Adapter Contract (Publisher Service → Platform Adapter)

| Method | Arguments | Returns | Behavior |
|--------|-----------|---------|----------|
| `publish` | `platform`, `accountId`, `contentPayload`, `idempotencyKey` | `{ platformPostId, publishedAt, permalink, rawResponse }` | Creates a post. Blocks until platform accepts or rejects. |
| `uploadMedia` | `platform`, `accountId`, `mediaStream`, `mimeType`, `idempotencyKey` | `{ platformMediaId, processingStatus }` | Uploads binary media. Some platforms return a processing state requiring polling. |
| `pollMediaStatus` | `platform`, `accountId`, `platformMediaId` | `{ status: 'ready' \| 'pending' \| 'failed', error? }` | Required for Instagram and TikTok async media processing. |
| `deletePost` | `platform`, `accountId`, `platformPostId` | `{ success: boolean }` | Best-effort deletion. Some platforms (e.g., X) allow this; others (LinkedIn) may not support deletion via API. |
| `getAccountMetadata` | `platform`, `accountId` | `{ username, profileId, scopes }` | Validates token and fetches current account handle/page ID. |

#### External API Surfaces

- **Meta (Facebook & Instagram)**: Graph API `v18.0+`. Instagram publishing requires a two-phase `media` → `media_publish` flow for carousel and Reel content. Uses `POST` multipart for video. Rate limits returned in `x-business-use-case-usage` headers.
- **X (Twitter) API v2**: `POST /2/tweets`. Media uploads use the legacy v1.1 `media/upload` endpoint with `INIT`, `APPEND`, `FINALIZE` chunked flow. Strict character limits and deep reply/threading rules.
- **LinkedIn REST API**: `POST /v2/ugcPosts` or `posts` (versioned). Video uploads require a three-step resumable upload protocol (register → upload → finalize). Author URN must be prefixed correctly.
- **TikTok Research / Publish API**: Direct post and pull-from-URL modes. OAuth 2.0 with scope-based feature gating. Video status polling is mandatory before publishing.
- **YouTube Data API v3**: `videos.insert` with resumable upload protocol. Quota cost is high (e.g., 1600 units per video upload). Requires `part=snippet,status` parameters.

### Data It Owns

`platform_apis` holds no data in the operational MongoDB or Redis stores. It is the **authoritative external source** for:

- **Platform Canonical IDs**: `instagram_media_id`, `tweet_id`, `linkedin_share_urn`, `tiktok_item_id`, `youtube_video_id`. These are generated by the provider and stored by our `mongodb_ops` content metadata collection only as cached references.
- **Platform Media Handles**: Transient IDs such as Twitter `media_id_string` or Meta `media_fbid`, which must be retained until the final post is published.
- **Published State**: The ground truth of whether a post is live, removed, or shadow-banned exists solely on the provider. Our system tracks `publishedAt` and `platformPostId` as a mirror.
- **Token Validity & Scopes**: The provider unilaterally decides when an access token is expired, revoked, or insufficiently scoped.

### Failure Modes

| Failure | Manifestation | System Impact | Mitigation |
|---------|---------------|---------------|------------|
| **Rate Limit (HTTP 429)** | `x-ratelimit-remaining: 0`, `Retry-After` header | Job worker stalls; queue backlog builds. | `rate_limiter` token bucket pauses the account/platform pair; job is nacked back to `redis_streams_queue` with delay. |
| **Auth Expiry (HTTP 401/403)** | Invalid token, insufficient scope, user revoked access | Publish job fails permanently for that account. | `publisher_service` signals `auth_service` to mark connection as `NEEDS_RECONNECT`; user notified. |
| **Content Policy Rejection (HTTP 400/403)** | Copyright match, unsupported format, text flagged | Terminal job failure. | Error captured in MongoDB job log; no retry. User receives failure notification with platform error code. |
| **API Outage (HTTP 5xx / Timeout)** | Gateway errors, TCP timeouts, TLS failures | Wasted worker cycles and risk of duplicate posts. | `circuit_breaker` opens after threshold; subsequent requests fail fast for the platform. Jobs retry later via queue dead-letter with backoff. |
| **Async Media Processing Failure** | Platform accepts upload but `processingStatus` returns `failed` | Post cannot complete. | `job_worker` polls `pollMediaStatus`; on failure, aborts publish and surfaces media error to user. |
| **Idempotency Window Expiry** | Platform rejects duplicate key after TTL | Risk of double-posting if client retries. | `publisher_service` checks MongoDB for existing `platformPostId` before retrying a publish call. |
| **Partial Batch Failure** | Meta batch API returns mixed `200` and `400` results | Inconsistent state if not handled per-item. | Adapter avoids batch publish for critical posts; uses individual requests to ensure per-item ack/nack. |
| **Breaking Change / Deprecation** | Endpoint removed, required field added | Sudden spike in publish failures. | Adapter pins to explicit API versions; alerts trigger when platform announces sunset; fallback to manual intervention. |

### Scaling Considerations

- **Hard External Ceilings**: Internal horizontal scaling of `job_worker` pods is ineffective if the aggregate outbound request volume exceeds per-app or per-account platform quotas. The `rate_limiter` must gate egress so that workers do not spawn redundant connections that will be throttled or banned.
- **Connection Reuse**: Adapters must use keep-alive HTTPS agents and HTTP/2 multiplexing where supported (e.g., Graph API) to avoid TLS handshake overhead across thousands of daily posts.
- **Regional Egress Latency**: Workers should be deployed in regions close to platform API gateways (e.g., US-East for Meta, US-West for X) or use egress proxies to minimize TCP round-trip time during chunked uploads.
- **Expensive Quota Platforms**: YouTube Data API quota is global per API key per day. Upload scheduling must be prioritized; non-essential read operations (e.g., metadata fetches) should be cached in `redis_cache` or avoided.
- **Resumable Uploads & Worker Blocking**: Large video uploads to LinkedIn or YouTube can exceed 60 seconds. The adapter should support resumable sessions so that transient worker restarts do not require re-uploading from byte zero. Long uploads should be delegated to a sidecar or async HTTP client to free the job worker thread.
- **Per-Account Sub-Partitioning**: A single user with hundreds of posts should not starve other users sharing the same platform app credentials. The `rate_limiter` enforces per-account fairness queues within the global app limit.
- **API Version Pinning & Adapter Drift**: Each platform adapter must declare its pinned API version in configuration. During platform-mandated migrations, multiple adapter versions may need to run concurrently (blue/green) to prevent mass publish failures.
