# ADR-003: Media Storage Strategy

## Status

Accepted

## Context

The platform must persist user-generated photos and videos for automated publishing across Instagram, Twitter, Facebook, and other social networks. Each platform enforces distinct technical requirements—maximum file size, aspect ratio, resolution, codec, and container format—that necessitate storing both the original upload and one or more processed derivatives. The chosen architecture uses Node.js/Express for the backend, MongoDB as the primary database, and a dedicated S3-compatible object store for binary assets.

Key requirements driving this decision:
- Accept original uploads up to 100 MB for video and 10 MB for images without blocking the HTTP request thread.
- Generate platform-optimized variants asynchronously.
- Serve secure previews to the web dashboard.
- Support post rescheduling and retries without requiring users to re-upload.
- Prevent storage bloat and unbounded cost growth as the user base scales.

Storing media binaries directly in MongoDB (GridFS) was rejected because it would inflate replica set oplog traffic and degrade query performance. Relying solely on S3 object tags for metadata was rejected because it makes indexed queries by user, job status, or publish schedule impossible at scale.

## Decision

Adopt a **dual-tier storage model**:

1. **S3-compatible Object Storage** (`object_storage`) is the immutable source of truth for all binary data: original uploads and every processed derivative.
2. **MongoDB** owns the canonical metadata, processing state, platform variant mappings, and lifecycle timestamps in the `media_assets` collection.

The `media_service` is the sole component authorized to write to the media bucket. All other services (`post_service`, `platform_connector`, `scheduler_service`) access media through pre-signed URLs or proxy streams provided by `media_service`.

### Storage Layout & Naming Convention

Bucket key structure:
```
users/{userId}/media/{mediaId}/
  original-{timestamp}.{ext}
  processed-{platform}-{timestamp}.{ext}
  thumbnail-{timestamp}.jpg
```

- `mediaId` is a MongoDB ObjectId generated by `media_service` at the start of the upload handshake.
- The `userId` prefix simplifies IAM policy scoping, cost allocation, and future bucket sharding if a single bucket approaches the provider’s object count limit.

### Metadata Schema (MongoDB)

Collection: `media_assets`

| Field | Type | Purpose |
|---|---|---|
| `_id` | ObjectId | Primary key; exposed as `mediaId` to clients |
| `userId` | ObjectId (Indexed) | Ownership and query isolation |
| `filename` | String | Original client file name |
| `mimeType` | String | Whitelist: `image/jpeg`, `image/png`, `video/mp4` |
| `checksum` | String | SHA-256 of the original upload for deduplication |
| `originalKey` | String | S3 key pointing to the untouched upload |
| `status` | String | Enum: `pending` → `processing` → `ready` \| `failed` |
| `variants` | Embedded Array | One element per derivative: `{ platform, s3Key, width, height, codec, sizeBytes, contentType, createdAt }` |
| `metadata` | Object | Video: `{ duration, fps, bitrate }`. Image: `{ exifOrientation, colorSpace }` |
| `createdAt` | Date | Upload completion time |
| `expiresAt` | Date | Drives TTL-based cleanup |

Indexes:
- `{ userId: 1, createdAt: -1 }` — dashboard pagination.
- `{ userId: 1, checksum: 1 }` — duplicate detection within a user scope.

### Upload Interface

1. The client requests a pre-signed upload URL from `POST /api/v1/media/upload-url`.
   - Files ≤ 10 MB receive a single PUT URL.
   - Files > 10 MB receive a multipart upload ID and part URLs.
2. The client streams bytes directly to Object Storage, bypassing the Express.js API Gateway to avoid memory pressure.
3. The client notifies `media_service` via `POST /api/v1/media/complete` with the `uploadId`, `key`, and `checksum`.
4. `media_service` validates the S3 HEAD response, inserts a `media_assets` document with `status: pending`, and enqueues an Agenda.js job named `media.process`.

### Processing Pipeline

- **Trigger**: `scheduler_service` or an on-demand API call enqueues `media.process` into the Agenda.js queue.
- **Execution**: A stateless worker pod pulls the original from S3 to an ephemeral NVMe scratch disk.
- **Image processing**: Sharp (libvips) resizes, strips EXIF, and outputs JPEG or PNG according to per-platform specification sheets (e.g., Instagram 1080×1080, Twitter 1200×675).
- **Video processing**: FFmpeg transcodes to H.264/AAC MP4 using platform-specific bitrate caps. A 1080p60 source is capped at 8 Mbps for Instagram and 5 Mbps for Twitter. Audio is normalized to AAC 128 kbps stereo.
- **Output**: The worker writes each variant back to S3 under the `processed-{platform}` key, updates `media_assets.variants`, and sets `status: ready`.
- **Guardrails**: Maximum job wall-clock time is 15 minutes. The FFmpeg process is killed with `SIGKILL` after `min(5 × videoDuration, 15 minutes)`.

### Security & Access Control

- The S3 bucket is **private with all public access blocked**.
- `media_service` generates **time-limited pre-signed URLs** (15-minute expiry) for dashboard previews and for `platform_connector` downloads.
- When a platform supports direct URL ingestion, `platform_connector` requests a fresh pre-signed GET URL from `media_service` immediately before the platform API call; URLs are never cached by downstream services.
- CORS and bucket policies restrict referrer access to the platform’s known web domains.

### Data Retention & Lifecycle

| Tier | Retention Rule | Implementation |
|---|---|---|
| Originals | 90 days after the last associated post is published | S3 Lifecycle transitions to Glacier after 30 days; hard-deleted after 90 days |
| Processed variants | 30 days after successful publish | MongoDB TTL index on `expiresAt`; matching S3 Lifecycle delete rule |
| Failed uploads | 7 days | S3 Lifecycle prefix rule targeting keys where `status: failed` |
| Thumbnails | Same lifecycle as original | Deleted alongside the original by the garbage-collector job |

Cleanup is executed by a nightly Agenda.js job `media.garbageCollect`:
1. Query `media_assets` for documents where `expiresAt < now`.
2. Issue batched S3 `DeleteObjects` requests (max 1,000 keys per call).
3. Remove MongoDB documents in corresponding batches.

## Failure Modes

| Scenario | Impact | Mitigation |
|---|---|---|
| **Upload interruption** | Partial object in S3, orphan metadata record | Multipart uploads auto-abort incomplete uploads after 24 hours. Client retries with the same `mediaId`; checksum deduplication prevents duplicate documents. |
| **FFmpeg hang or corrupted input** | Worker stuck, temp disk exhaustion | Job timeout enforces `SIGKILL`. Temp files are wiped on container termination. Asset is marked `status: failed`; `notification_service` emails the user. |
| **Object Storage region outage** | Publish jobs cannot retrieve media | `media_service` returns HTTP 503 with `Retry-After: 300`. Agenda.js retries with exponential backoff (max 6 hours). Platform publishes are deferred until media is accessible. |
| **Corruption discovered post-upload** | Variant generation repeatedly fails | The original is moved to `quarantine/{userId}/{mediaId}/` via S3 Copy + Delete. `media_assets.status` is set to `quarantined`. The user is notified with the failure reason. |
| **Duplicate upload** | Wasted storage and compute | SHA-256 deduplication per `userId`. If a match is found, the existing `mediaId` is returned. A `refCount` field on the document tracks shared usage so the object is only deleted when all references are removed. |
| **Pre-signed URL expiry during publish** | Platform API returns 403 | `platform_connector` is forbidden from caching pre-signed URLs. It must request a fresh URL from `media_service` within 60 seconds of the platform API call. |

## Scaling Considerations

- **Read throughput**: Thumbnails and processed variants are fronted by a CDN with origin-pull from Object Storage. For dashboard previews, signed cookies at the edge offload pre-signed URL generation from `media_service` after an initial trust exchange.
- **Write throughput**: Multipart uploads allow horizontal scaling of file size without increasing memory on API Gateway pods. `media_service` API pods scale via Kubernetes HPA based on request latency; worker pods scale based on Agenda.js queue depth (`agendaJobs.count({ name: 'media.process', status: 'pending' })`).
- **CPU-bound processing**: Video transcoding is isolated to a dedicated node pool (tainted `media-workers`) to prevent starving latency-sensitive API pods. CPU limit is 4 cores per worker; concurrency is capped at 2 simultaneous FFmpeg jobs per pod.
- **Storage cost**: Originals transition to S3 Intelligent-Tiering or Glacier after 30 days because they are rarely accessed again. Processed variants remain in the hot tier for fast CDN origin-pull during the active publish window, then expire rapidly.
- **Database growth**: The `media_assets` collection is sharded by `userId` once the working set exceeds 10 TB. The embedded `variants` array is capped to one entry per active platform (≤ 10). If future requirements demand more derivatives, a separate `media_variants` collection will be introduced.
- **Cross-region replication**: Not implemented at launch. If disaster-recovery requirements change, original uploads will be replicated asynchronously to a secondary-region bucket, while MongoDB relies on the existing replica set topology in that region.
