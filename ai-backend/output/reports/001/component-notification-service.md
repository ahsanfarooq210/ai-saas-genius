## component-notification-service.md

## Responsibilities

The Notification Service is responsible for delivering user-facing alerts across email and push channels. It acts as a centralized dispatcher for all asynchronous communication generated by the platform's background job and publishing pipeline.

Specific duties include:

- **Dispatching email notifications** for post-publish outcomes (success or failure), OAuth token expiration, platform account disconnections, and quota limit warnings.
- **Delivering push notifications** via Firebase Cloud Messaging (FCM) and Apple Push Notification service (APNs) for time-sensitive events such as immediate publish failures or critical account issues.
- **Managing per-user notification preferences**, including channel opt-ins, quiet hours, and enabled notification types.
- **Rendering message templates** with contextual variables (e.g., `postId`, `platform`, `failureReason`) before delivery.
- **Tracking delivery state** by persisting send receipts, provider message IDs, bounce events, and permanent failures.
- **Enforcing rate limits** to prevent notification spam (e.g., max one email per 15 minutes for the same failure type per user).

## APIs / Interfaces

### Internal REST API

The service exposes an internal HTTP API consumed by the `agenda_worker` and `publisher_service`. All endpoints require an internal service-to-service authentication header (`X-Internal-Service-Key`).

#### `POST /v1/notifications/dispatch`

Dispatches one or more notifications.

**Request body:**
```json
{
  "notifications": [
    {
      "userId": "507f1f77bcf86cd799439011",
      "type": "publish_failure",
      "channels": ["email", "push"],
      "payload": {
        "postId": "post_123",
        "platform": "instagram",
        "errorCode": "RATE_LIMITED",
        "errorMessage": "API quota exceeded"
      },
      "idempotencyKey": "pubfail-507f1f77-20240115T120000Z"
    }
  ]
}
```

**Response:**
```json
{
  "accepted": 1,
  "rejected": 0,
  "notificationIds": ["notif_abc123"]
}
```

#### `GET /v1/notifications/preferences/:userId`

Retrieves the user's current notification preferences.

**Response:**
```json
{
  "userId": "507f1f77bcf86cd799439011",
  "channels": {
    "email": {
      "enabled": true,
      "address": "user@example.com"
    },
    "push": {
      "enabled": true,
      "tokens": [
        { "platform": "fcm", "token": "fcm_token_xyz", "updatedAt": "2024-01-10T10:00:00Z" }
      ]
    }
  },
  "quietHours": { "start": "22:00", "end": "07:00", "timezone": "America/New_York" },
  "enabledTypes": ["publish_success", "publish_failure", "account_disconnected"]
}
```

#### `PUT /v1/notifications/preferences/:userId`

Updates preferences. Validates email format and push token uniqueness.

### External Provider Interfaces

- **AWS SES / SendGrid REST API** for email transport.
- **FCM HTTP v1 API** for Android and web push.
- **APNs HTTP/2 API** for iOS push.
- **Provider Webhooks** (SES SNS, SendGrid Event Webhook) to ingest bounce, complaint, and delivery events.

## Data Owned

The service owns the following MongoDB collections:

### `notification_logs`

Immutable record of every notification dispatched.

| Field | Type | Description |
|-------|------|-------------|
| `_id` | ObjectId | Unique log identifier |
| `userId` | ObjectId | Recipient user reference |
| `type` | String | Event type (`publish_success`, `publish_failure`, `account_disconnected`, etc.) |
| `channel` | String | `email` or `push` |
| `payload` | Object | Serialized contextual data used for rendering |
| `status` | String | `pending`, `sent`, `failed`, `bounced`, `delivered` |
| `providerMessageId` | String | External provider's message ID |
| `idempotencyKey` | String | Prevents duplicate sends |
| `error` | Object | `{ code, message }` on failure |
| `createdAt` | Date | Log creation timestamp |
| `sentAt` | Date | Provider acceptance timestamp |

**Indexes:**
- `{ userId: 1, createdAt: -1 }` for user-facing notification history queries.
- `{ idempotencyKey: 1 }` with unique constraint for deduplication.
- TTL index on `createdAt` (90 days) for automatic log expiration.

### `notification_preferences`

Per-user configuration document.

| Field | Type | Description |
|-------|------|-------------|
| `userId` | ObjectId | Primary key |
| `channels.email` | Object | `{ enabled: Boolean, address: String, verified: Boolean }` |
| `channels.push` | Object | `{ enabled: Boolean, tokens: Array }` |
| `quietHours` | Object | `{ start: String, end: String, timezone: String }` |
| `enabledTypes` | Array[String] | Whitelist of notification types the user accepts |
| `updatedAt` | Date | Last modification timestamp |

### `notification_templates`

Versioned templates for each notification type and channel.

| Field | Type | Description |
|-------|------|-------------|
| `type` | String | Notification type key |
| `channel` | String | `email` or `push` |
| `subjectTemplate` | String | Handlebars template for email subject |
| `bodyTemplate` | String | Handlebars template for body/push content |
| `variables` | Array[String] | Required payload keys |
| `version` | Number | Template revision |
| `default` | Boolean | Whether this is the active default |

## Failure Modes

| Failure | Impact | Mitigation |
|---------|--------|------------|
| **Email provider outage (SES/SendGrid)** | Notifications queue up; users are unaware of publish failures. | Circuit breaker on provider client. Failed dispatches remain in `notification_logs` with `status: pending` and a `retryAfter` timestamp. A background polling job retries with exponential backoff (max 6 hours). |
| **Push token invalid / unregistered** | APNs/FCM returns 410 or `NotRegistered`. | Capture error, remove stale token from `notification_preferences.channels.push.tokens`, and disable push if no valid tokens remain. Do not retry. |
| **Template rendering error (missing variable)** | Notification cannot be constructed. | Fallback to a generic default template for the type/channel. Log the schema mismatch and emit an error metric to trigger an operational alert. |
| **Rate limit hit (provider or internal)** | 429 responses from FCM/SES or internal per-user throttle exceeded. | Internal rate limiter (token bucket per user per type). Provider 429s trigger the same retry queue as outages. |
| **Notification service unavailable** | `agenda_worker` and `publisher_service` calls fail synchronously. | Calling services must treat notification dispatch as best-effort. They should use short timeouts (≤ 3s) and continue job execution without halting the publish pipeline. |
| **MongoDB write conflict on preferences** | Concurrent updates from user action and token cleanup. | Optimistic locking with `updatedAt` version check; retry once on `MongoWriteConflict`. |

## Scaling Considerations

- **Stateless horizontal scaling:** The service runs as a stateless Node.js/Express process. Any instance can handle any request. Scale out based on CPU and custom metrics (notification dispatch queue depth).
- **Channel isolation:** Under high load, email and push dispatch can be partitioned into separate worker pools or message topics to prevent slow SMTP handshakes from blocking push latency.
- **Batching:** Email provider APIs support batch submissions. The service should accumulate pending email notifications for the same user over a 30-second window where possible to reduce API call volume.
- **Connection pooling:** Maintain HTTP/2 connection pools to FCM and APNs, and SMTP connection pools to SES, to avoid connection establishment overhead under burst load.
- **Idempotency:** All dispatch requests must include an `idempotencyKey` generated by the caller (e.g., `publisher_service`). The service deduplicates within a 24-hour window to prevent duplicate alerts during caller retries.
- **Webhook ingestion:** Provider webhook endpoints must be rate-limited and authenticated to prevent abuse. Ingested events are written to a capped collection or processed via a change stream if high volume is expected.
- **Query performance:** The `notification_logs` collection uses a compound index on `{ userId: 1, createdAt: -1 }` to support paginated notification history lookups without table scans.
