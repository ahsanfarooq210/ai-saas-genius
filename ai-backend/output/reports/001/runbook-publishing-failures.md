# Runbook: Publishing Failures

## Scope
This runbook covers detection, diagnosis, and recovery for failures in the automated content publishing pipeline. It spans the `agenda_worker`, `publisher_service`, `platform_api_clients`, `scheduler_service`, `content_service`, `media_service`, `auth_service`, and `token_store`. Failures may manifest as missed post schedules, duplicate publishes, platform API rejections, or stuck background jobs.

---

## Severity Classification

| Level | Criteria | Response Time |
|-------|----------|---------------|
| **SEV 1** | Zero publishes succeeding across all users and platforms; or data loss risk | 5 min |
| **SEV 2** | All publishes failing for a single platform (e.g., Instagram); or >25% user base affected | 15 min |
| **SEV 3** | Isolated user account failures, recurring token expiry, or individual stuck jobs | 1 hour |
| **SEV 4** | Intermittent retry-able errors with eventual success; minor timezone drift | 4 hours |

---

## Symptoms & Detection

- **Metrics:** `publisher_service` error rate > 1% over 5 minutes; `agenda_worker` job lag > 10 minutes.
- **Database:** `agendaJobs` collection shows documents in `failed` state or `lockedAt` timestamps older than the job’s `lockLifetime` without a corresponding `lastFinishedAt`.
- **Logs:** `platform_api_clients` emitting HTTP `4xx`/`5xx` status codes; `publisher_service` logging `PublishError`.
- **Notifications:** Spike in `publish_failed` events emitted to the `notification_service`.
- **User Reports:** Scheduled posts not appearing on connected social feeds.

---

## Failure Modes & Diagnostics

### 1. Agenda Worker Job Stalls
**Indicators:** Jobs remain in `locked` state; `lastRunAt` is stale; worker container restart loops.
**Root Causes:** Worker process OOM-killed; MongoDB replica set failover preventing lock release; unhandled exception in job processor; Node.js event loop blocked by synchronous media encoding.
**Diagnostics:**
```javascript
// Find jobs stuck beyond lock lifetime (default 5 min)
db.agendaJobs.find({
  lockedAt: { $lt: new Date(Date.now() - 10*60*1000) },
  lastFinishedAt: { $exists: false }
})
```
Check `agenda_worker` pod memory/CPU throttling and MongoDB primary connectivity.

### 2. Publisher Service Platform Rejections
**Indicators:** `publisher_service` logs show `statusCode: 400|403|429|500` from platform endpoints.
**Root Causes:** Malformed payload generated by `content_service`; platform rate limiting; breaking API schema change; video duration exceeds platform limit.
**Diagnostics:**
- Inspect `publisher_service` log field `platformResponse` for error details and `x-ratelimit-remaining` headers.
- Validate the outbound payload against the `platform_api_clients` request schema for the target network.
- Check `content_service` draft document in MongoDB for `caption`, `hashtags`, and `mediaType` compliance.

### 3. OAuth Token Invalidity
**Indicators:** HTTP `401` or `403` from platform APIs; `TokenRefreshError` in `auth_service` logs.
**Root Causes:** User revoked application permissions in platform settings; refresh token expired (e.g., Instagram long-lived tokens > 60 days); `token_store` decryption key mismatch after rotation.
**Diagnostics:**
- Query the `token_store` (MongoDB encrypted collection) for the user’s `platformConnection`:
  ```javascript
  db.platformConnections.findOne({ userId: "<uid>", platform: "instagram" })
  ```
- Verify `refreshTokenExpiresAt` and `lastRefreshedAt`.
- Attempt a manual refresh via `auth_service` internal endpoint `POST /internal/oauth/refresh`.

### 4. Media Asset Unreachable
**Indicators:** Platform API returns media download errors or timeout; `media_service` shows `processingStatus: failed`.
**Root Causes:** Object storage pre-signed URL expired before publish execution; CDN edge 404; `media_service` thumbnail generation failure leaving video without a poster frame required by the platform.
**Diagnostics:**
- Confirm `media_service` record status is `completed` in MongoDB `media` collection.
- `curl -I <mediaUrl>` from a public egress point (platform crawlers cannot access internal VPC URLs).
- Verify object existence in `object_storage` bucket via S3-compatible `HeadObject` API.

### 5. Scheduler Misconfiguration & Timezone Drift
**Indicators:** Posts publish at incorrect local times; jobs missing on weekends; duplicate `agendaJobs` entries for the same `contentId`.
**Root Causes:** Invalid cron syntax in `scheduler_service` generated job definitions; server system clock not synchronized to UTC; daylight saving time boundary not handled by Agenda.js `repeatTimezone`.
**Diagnostics:**
- Inspect the job definition:
  ```javascript
  db.agendaJobs.find({ "data.contentId": "<id>" }, { nextRunAt: 1, repeatInterval: 1, repeatTimezone: 1 })
  ```
- Validate `repeatInterval` cron string with `node-cron` parser.
- Check worker node `timedatectl` or container `TZ` environment variable.

### 6. MongoDB State Desync
**Indicators:** Agenda reports job success but `posts` collection shows `status: draft`; or job fails while `posts` shows `status: published`.
**Root Causes:** Transaction rollback failure between `publisher_service` write and Agenda job completion; MongoDB write concern `w: 1` lost acknowledgment during failover.
**Diagnostics:**
- Correlate `agendaJobs._id` with `posts.publishJobId`.
- Check MongoDB replica set lag and oplog window.

## Recovery Procedures

### Immediate Mitigation (SEV 1 / SEV 2)
1. **Halt Job Pickup:** Set `agenda_worker` environment variable `PUBLISHING_ENABLED=false` and rolling-restart workers to stop consuming new jobs without killing in-flight requests.
2. **Unlock Stuck Jobs:** Reset locks to allow healthy workers to reclaim:
   ```javascript
   db.agendaJobs.updateMany(
     { lockedAt: { $lt: new Date(Date.now() - 15*60*1000) } },
     { $set: { lockedAt: null, lastModifiedBy: "runbook-publishing-failures" } }
   )
   ```
3. **User Communication:** Trigger `notification_service` to queue delay notices for all posts scheduled within the next 4 hours.

### Per-Failure Recovery

**Agenda Worker Crash / OOM**
- Scale workers horizontally: `kubectl scale deployment agenda-worker --replicas=6`
- Increase `lockLifetime` from default `5m` to `10m` for video-heavy workloads.
- Ensure MongoDB connection pool size (`maxPoolSize`) exceeds total worker count.

**Platform Rate Limiting (429)**
- Pause retries for the affected platform by setting `platform_api_clients` config `circuitBreaker.enabled=true`.
- Read `Retry-After` header from platform response; reschedule the Agenda job explicitly:
  ```javascript
  const job = await agenda.jobs({ "data.contentId": "<id>" });
  job[0].schedule(new Date(Date.now() + retryAfterMs));
  await job[0].save();
  ```
- Do not retry blindly; respect platform windows (Instagram Basic Display: 200 calls/hour per user).

**Token Expiry / Revocation**
- If refresh token is valid: invoke `auth_service` to refresh and update `token_store` document atomically.
- If user revoked access: update `user_service` record `platformConnections.$.status` to `disconnected`; send `notification_service` email with re-auth deep link.
- Backfill: for failed jobs due to token issues, reset `failCount` to 0 and `nextRunAt` only after token is confirmed valid.

**Media Unavailable**
- If `object_storage` URL expired: call `media_service` internal endpoint `POST /internal/media/regenerate-url` with `mediaId` and TTL ≥ 48 hours.
- If processing incomplete: re-queue `media_service` processing job; block Agenda job execution until `processingStatus` is `completed` using a pre-publish hook.

**Content Validation Errors**
- Patch the `content_service` draft (truncate caption to platform limit, strip banned hashtags).
- Update `scheduler_service` to run validation rules against `platform_api_clients` schemas before job creation.

**Scheduler Timezone / Duplicate Jobs**
- Delete duplicates while preserving the earliest `nextRunAt`:
  ```javascript
  // Keep one, remove others
  db.agendaJobs.deleteMany({
    name: "publish-post",
    "data.contentId": "<id>",
    _id: { $nin: [ <keepId> ] }
  })
  ```
- Force UTC on all worker nodes and Agenda `repeatTimezone` fields.

## Verification Checklist

- [ ] Target `agendaJobs` document shows `lastFinishedAt` within the scheduled window and `failCount: 0`.
- [ ] `posts` collection record reflects `status: published`, `publishedAt` timestamp, and `platformPostId` returned by the API.
- [ ] `publisher_service` log contains `event: publish_success` with `duration_ms` and `platform: <name>`.
- [ ] `notification_service` emitted `publish_success` to the user within 30 seconds of completion.
- [ ] No duplicate `platformPostId` exists for the same `contentId` (idempotency check).

---

## Prevention & Scaling Considerations

- **Idempotency:** `publisher_service` must generate a deterministic idempotency key (hash of `userId`, `contentId`, and `scheduledAt`) and pass it to `platform_api_clients`. On retry, platforms that support idempotency keys will reject duplicates.
- **Circuit Breakers:** `platform_api_clients` should trip after 5 consecutive errors for a single platform, returning `PlatformUnavailableError` to the worker for a 2-minute cooldown.
- **Proactive Token Refresh:** A nightly Agenda job should refresh OAuth tokens via `auth_service` when `expiresAt < now + 14 days`, updating `token_store` before user-facing publishes are affected.
- **Media URL TTL:** Object storage pre-signed URLs must have a TTL ≥ 48 hours to survive job delays and retries.
- **Job Timeouts:** Set Agenda `lockLifetime` to at least 5 minutes for image posts and 15 minutes for video posts to accommodate platform upload latency.
- **Observability:** Propagate OpenTelemetry `traceparent` from `agenda_worker` → `publisher_service` → `platform_api_clients` so a single trace captures the full publish attempt including media fetch and API response.

## Escalation

| Severity | Action |
|----------|--------|
| **SEV 1** | Page on-call engineer and platform team lead. Open war room in `#incidents`. |
| **SEV 2** | Notify platform-specific SME (e.g., Instagram API specialist). |
| **SEV 3** | Create ticket for `auth_service` or `content_service` team; monitor for 24 hours. |
| **SEV 4** | Log in observability backlog; review at next sprint planning. |
