# component-object-storage

## Responsibilities

The object storage layer provides durable, S3-compatible blob storage for all media assets in the social media automation platform. Its specific responsibilities include:

- **Original Asset Preservation**: Storing immutable copies of user-uploaded photos and videos immediately upon ingestion from the `media_service`, ensuring that source files are never overwritten or lost during processing.
- **Processed Derivative Storage**: Retaining transcoded video variants, compressed images, and generated thumbnails produced by the `media_service` to meet platform-specific requirements (e.g., Instagram 1080×1080, TikTok 1080×1920).
- **Secure Asset Delivery**: Supporting time-limited, cryptographically signed read URLs (`presigned URLs`) so that web and mobile clients, as well as downstream `publisher_service` workers, can retrieve media without exposing buckets to public access.
- **Lifecycle and Compliance Management**: Enforcing retention policies, automatic expiration of temporary multipart upload fragments, and storage-class transitions (e.g., from Standard to Infrequent Access) for archived campaign assets.
- **Access Isolation**: Enforcing least-privilege IAM/bucket policies so that only the `media_service` instance role and designated background workers can perform write operations.

## APIs and Interfaces

The component exposes a standard S3-compatible REST API. The `media_service` and administrative tooling interact with it using the AWS SDK for JavaScript v3 (`@aws-sdk/client-s3`). Key operations include:

| Operation | SDK Command | Purpose |
|-----------|-------------|---------|
| **Object Upload** | `PutObject` / `CreateMultipartUpload` + `UploadPart` + `CompleteMultipartUpload` | Stores original uploads and processed derivatives. Multipart upload is mandatory for video files > 100 MB. |
| **Object Retrieval** | `GetObject` | Used by CDN origin pull and by the `media_service` when generating presigned read URLs for the `publisher_service`. |
| **Metadata Check** | `HeadObject` | Verifies existence, `ETag`, and `Content-Type` without downloading the full payload; used before scheduling a post to confirm media is ready. |
| **Object Deletion** | `DeleteObject` / `DeleteObjects` | Removes media during user account deletion, content moderation takedowns, or garbage collection of orphaned temp files. |
| **Presigned URL Generation** | `GetObjectCommand` + `getSignedUrl` | Produces short-lived HTTPS URLs (default 15 minutes) returned to clients via the `media_service` API for direct browser/mobile playback. |
| **List & Query** | `ListObjectsV2` | Restricted to admin/maintenance tasks; not used in hot-path request handling. |

**Interface Constraints**
- All writes require `x-amz-meta-media-id` and `x-amz-meta-user-id` headers to correlate blobs with MongoDB records.
- Bucket CORS rules allow `GET` and `PUT` only from the platform's origin domains if direct browser-to-storage uploads are enabled.
- Pre-signed `PUT` URLs can be generated by the `media_service` to let clients upload directly to object storage, offloading the Express.js API gateway from binary traffic.

## Data Ownership and Storage Model

Object storage owns the **raw bytes** of all media; metadata of record (captions, hashtags, processing status, platform publish history) remains in MongoDB.

**Bucket Layout**
```
s3://{bucket}/
├── uploads/
│   └── {userId}/
│       └── {mediaId}/
│           └── original.{ext}          # Immutable source file
├── processed/
│   └── {userId}/
│       └── {mediaId}/
│           ├── thumbnail.webp        # 300x300 preview
│           ├── image_optimized.jpg   # Platform-agnostic compressed image
│           ├── video_1080p.mp4       # Highest quality transcode
│           └── video_720p.mp4        # Fallback transcode
└── temp/
    └── {uploadId}/                   # Multipart upload staging
```

**Object Metadata (S3 user metadata)**
- `x-amz-meta-media-id`: Links the blob to the `Media` document in MongoDB.
- `x-amz-meta-user-id`: Enforces ownership boundaries and simplifies garbage collection scans.
- `x-amz-meta-content-type`: Normalized MIME type (`image/jpeg`, `video/mp4`).
- `x-amz-meta-checksum-sha256`: Client-computed hash for end-to-end integrity verification.

**Storage Classes**
- **STANDARD**: Active campaign assets and originals.
- **STANDARD_IA**: Completed campaign media older than 90 days.
- **GLACIER/DEEP_ARCHIVE**: Long-term retention for legal/compliance (if required).

## Failure Modes and Mitigations

| Failure Mode | Impact | Mitigation |
|--------------|--------|------------|
| **Incomplete Multipart Upload** | Orphaned parts accumulate after network drops during large video uploads, inflating storage costs. | Bucket lifecycle rule to abort incomplete multipart uploads after 7 days. |
| **IAM Policy Misconfiguration** | `media_service` loses write access; user uploads fail. | Infrastructure-as-Code (Terraform/Pulumi) for policy definitions; automated IAM policy drift detection. |
| **Object Corruption / Hash Mismatch** | `ETag` or SHA-256 metadata does not match the uploaded payload; downstream publishing delivers broken media. | `media_service` validates checksums after `CompleteMultipartUpload` and before marking the MongoDB media record as `ready`. |
| **Presigned URL Expiry** | `publisher_service` or client receives `403 Forbidden` if a URL expires mid-download or during a delayed publish job. | Use 1-hour expiry for backend workers (`publisher_service`) and 15-minute expiry for client previews; workers refresh URLs via the `media_service` before platform API calls. |
| **Storage Exhaustion / Quota** | User uploads exceed provisioned capacity or budget, causing all platform uploads to fail. | Per-user soft quotas enforced in `media_service`; CloudWatch/alerts on total bucket size; auto-expiration of `temp/` prefix. |
| **Cross-Region Unavailability** | Primary region outage makes scheduled posts un-publishable. | Cross-region replication (CRR) for `processed/` assets; `publisher_service` fails over to replica bucket in DR region. |
| **Orphaned Objects** | MongoDB media record is deleted (user action), but S3 objects remain, leading to unbilled storage growth. | Nightly garbage-collection job listing `x-amz-meta-media-id` in S3 and diffing against MongoDB `media` collection; deletes unmatched objects after a 48-hour grace period. |

## Scaling Considerations

- **Video Throughput**: Social media campaigns generate high volumes of large video files. Multipart uploads must be parallelized (minimum 5 MB per part, up to 10,000 parts) to saturate available bandwidth. The `media_service` should stream uploads directly from the API gateway request stream to S3 without buffering the entire file in Node.js memory.
- **CDN Offload**: All client reads and `publisher_service` downloads should route through a CDN (e.g., CloudFront) backed by the S3 origin. This reduces S3 request costs and improves global latency for thumbnail previews in the web dashboard.
- **Hot Prefix Avoidance**: S3 partitions performance by prefix. Using `{userId}` as the first path segment distributes load, but high-traffic users could still create hot partitions. For extreme scale, inject a random 2-character hex prefix before `userId` (e.g., `uploads/a3/{userId}/...`).
- **Egress Optimization**: Publishing to platforms like Instagram and TikTok requires the platform to fetch the media URL. Serving these URLs from a CDN edge rather than directly from S3 avoids costly S3 egress charges, especially for viral high-resolution video content.
- **Request Rate Limits**: While S3 scales to thousands of TPS per prefix, S3-compatible alternatives (MinIO, Wasabi, DigitalOcean Spaces) may have lower per-bucket or per-prefix rate limits. Monitor `503 Slow Down` errors and implement exponential backoff with jitter in the `media_service` S3 client.
- **Lifecycle Automation**: As the platform grows, automate transition of `processed/` objects older than 90 days to Infrequent Access, and originals older than 1 year to Glacier, to control storage spend without manual intervention.
