## S3 Storage

## Overview
S3 Storage provides the platform’s primary object persistence layer for all media assets. It durably stores original user uploads (photos and videos) as well as platform-optimized renditions produced by the Media Service. The storage layer is consumed exclusively by the Media Service for writes and by the CDN for origin reads. No other service writes directly to S3.

## Responsibilities
- **Durable binary persistence** — Store original and processed media files with 99.999999999% durability guarantees.
- **Rendition storage** — Retain platform-specific outputs (e.g., Instagram 1080×1080 JPEG, TikTok 1080×1920 MP4, story thumbnails) generated by the Media Service.
- **Secure upload target** — Support server-side and direct-to-S3 client uploads via pre-signed URLs to minimize Node.js backend bandwidth.
- **Origin for CDN** — Serve as the authoritative content source for the CDN; no public bucket access is permitted.
- **Encryption & access control** — Enforce server-side encryption (SSE-S3 or SSE-KMS) and least-privilege IAM policies.
- **Lifecycle & cost management** — Automate transition of processed media to cheaper storage classes and clean up incomplete multipart uploads.

## APIs / Interfaces
The Media Service interacts with S3 using the AWS SDK for JavaScript v3 (`@aws-sdk/client-s3`).

| Operation | SDK Command | Purpose |
|-----------|-------------|---------|
| **Put Object** | `PutObjectCommand` | Upload small images (< 100 MB) and processed renditions from Media Service workers. |
| **Initiate Multipart Upload** | `CreateMultipartUploadCommand` | Begin resumable upload for large raw videos. |
| **Upload Part** | `UploadPartCommand` | Stream video chunks during multipart upload. |
| **Complete Multipart Upload** | `CompleteMultipartUploadCommand` | Finalize video assembly after all parts are uploaded. |
| **Abort Multipart Upload** | `AbortMultipartUploadCommand` | Cancel and delete fragments on processing failure or client timeout. |
| **Generate Pre-signed URL** | `getSignedUrl` (`@aws-sdk/s3-request-presigner`) | Issue time-limited `PUT` URLs for direct browser/app uploads; issue time-limited `GET` URLs for temporary private access. |
| **Delete Object** | `DeleteObjectCommand` | Remove originals or renditions when a user deletes content or when lifecycle jobs expire old media. |
| **Head Object** | `HeadObjectCommand` | Verify object existence, size, and `ETag` before triggering downstream processing. |

**Interface constraints**
- All traffic is forced over HTTPS.
- Bucket policies block all public access; read access is restricted to the Media Service IAM role and the CDN origin-access identity (OAI) / origin-access control (OAC).
- Pre-signed `PUT` URLs are scoped to a specific key and `Content-Type`, with an expiration of 15 minutes.

## Data Ownership & Object Layout
S3 owns the binary object content only. Descriptive metadata (file name, upload timestamp, processing status, platform variants, user ownership) is authoritative in MongoDB and managed by the Media Service.

**Bucket key prefix schema**

```
s3://{media-bucket}/
  originals/
    {userId}/
      {mediaUuid}/
        original.{ext}                 # User-uploaded source file
  processed/
    {userId}/
      {mediaUuid}/
        {platform}-{variant}.{ext}     # e.g., instagram-feed-1080x1080.jpg
        {platform}-{variant}.{ext}     # e.g., tiktok-video-720p.mp4
  temp/
    {uploadUuid}/
      {partNumber}                     # Multipart upload fragments (transient)
```

- `{mediaUuid}` and `{uploadUuid}` are random v4 UUIDs to ensure prefix partitioning and avoid hot spots.
- Object metadata headers (`x-amz-meta-media-id`, `x-amz-meta-user-id`) are written at upload time for coarse-grained tracing.

## Failure Modes & Mitigations
| Failure | Impact | Mitigation |
|---------|--------|------------|
| **Multipart upload never completes** | Orphaned parts accumulate, incurring storage costs. | Lifecycle rule to abort incomplete multipart uploads after 1 day. |
| **Upload succeeds but DB record fails** | “Orphaned” object with no MongoDB reference. | Nightly sweeper job (Agenda.js) lists `originals/` prefixes and reconciles against `Media` collection; unreferenced objects older than 7 days are deleted. |
| **Pre-signed URL expiry during client upload** | Large video upload fails mid-stream. | Use multipart pre-signed URLs (one per part) with 15-minute part expiry; client retries failed parts rather than restarting the whole file. |
| **S3 throttling (503 Slow Down)** | High concurrency on a single prefix degrades upload throughput. | UUID-based prefixes distribute load; monitor `5xx` errors and implement SDK retry with exponential backoff. |
| **Accidental object deletion** | Permanent loss of original media. | Enable bucket versioning and MFA delete; processed renditions can be regenerated from originals. |
| **Bucket misconfiguration / public access** | Media leakage or unauthorized deletion. | Block Public Access setting enabled at account and bucket level; IAM policies scoped to service roles only; quarterly configuration audits. |
| **Region-level outage** | Media unavailable for publishing and CDN origin pulls. | Evaluate Cross-Region Replication (CRR) for the `originals/` prefix if RTO/RPO requirements mandate it; CDN caching shields against short outages. |

## Scaling Considerations
- **Automatic horizontal scaling** — S3 partitions automatically, and the UUID prefix scheme ensures requests are distributed across the partition space without requiring sequential hash prefixes.
- **Large video handling** — All raw video uploads > 100 MB must use multipart upload with a minimum part size of 5 MB (except final part). The Media Service streams parts directly from the request buffer to S3 to avoid local disk exhaustion on worker nodes.
- **Cost optimization** — Processed media that is only accessed during publishing can be transitioned to S3 Intelligent-Tiering or S3 Standard-IA after 30 days. Temporary fragments in `temp/` expire automatically.
- **Request cost control** — The CDN sits in front of S3 for all GET/HEAD traffic. Direct S3 origin fetches are limited to cache misses and invalidation events. This reduces S3 request charges and latency.
- **Monitoring** — Track `AllRequests`, `4xxErrors`, `5xxErrors`, `TotalRequestLatency`, and storage class distribution via CloudWatch (or equivalent). Alert on spikes in `DeleteObject` or `PutObject` rates that deviate from expected publishing patterns.
