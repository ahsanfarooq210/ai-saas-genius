## WebSocket Gateway

### Responsibilities

- **Maintain persistent bidirectional connections** between authenticated dashboard clients and the backend, enabling real-time updates without polling the REST API.
- **Deliver job lifecycle events** generated by Agenda.js and the Job Service, including state transitions: `queued`, `preparing`, `publishing`, `published`, `failed`, and `retrying`.
- **Push account health alerts** from the Notification Service, such as OAuth token expiration warnings, platform rate-limit hits, and disconnected social accounts.
- **Broadcast media processing progress** from the Media Service (e.g., upload completion, transcoding stage, CDN propagation status) to users actively viewing the content library.
- **Synchronize cross-server state** using Redis so that horizontally scaled gateway nodes can route messages to the correct socket regardless of which instance a client lands on.

### APIs and Interfaces

#### Connection Handshake
- **Endpoint:** `wss://api.platform.com/ws/v1`
- **Transport:** WebSocket (native `ws` or Socket.io) over TLS.
- **Authentication:** JWT passed as a query parameter `?token=<jwt>` during the upgrade handshake. The gateway validates the token against the Auth Service before elevating the HTTP connection.
- **Heartbeat:** Server issues `ping` frames every 30 seconds; clients must respond with `pong` within 10 seconds or the socket is terminated and the Redis presence entry is purged.

#### Inbound Client Messages
| Event | Payload | Behavior |
|---|---|---|
| `subscribe_user` | `{ userId: string }` | Joins the socket to a Redis-backed room keyed by `user:{userId}`. The client receives all user-level notifications and job updates. |
| `subscribe_job` | `{ jobId: string }` | Joins a transient room `job:{jobId}` to receive granular status events for a specific Agenda.js job. Unsubscribes automatically when the job reaches a terminal state. |
| `subscribe_account` | `{ accountId: string }` | Joins `account:{accountId}` to receive platform-specific alerts (e.g., Instagram token refresh failures). |
| `ack_notification` | `{ notificationId: string }` | Sends a delivery acknowledgment to the Notification Service so the unread badge can be cleared. |

#### Outbound Server Events
| Event | Payload | Trigger |
|---|---|---|
| `job.status` | `{ jobId, status, agendaJobId, platform, scheduledAt?, completedAt? }` | Job Service transitions an Agenda.js job through its state machine. |
| `job.failure` | `{ jobId, errorCode, message, retryCount, maxRetries, nextRunAt? }` | Publish Service returns a non-retryable error or max retries exceeded. |
| `media.progress` | `{ mediaId, stage: 'uploaded' \| 'optimized' \| 'cdn_ready', percent }` | Media Service finishes a processing step. |
| `account.warning` | `{ accountId, platform, severity, code: 'TOKEN_EXPIRING' \| 'DISCONNECTED' \| 'RATE_LIMITED', actionRequired: boolean }` | Auth Service or Publish Service detects account health issues. |
| `notification` | `{ id, type, title, body, createdAt, metadata }` | Notification Service emits a user-facing alert. |

#### Internal Contracts
- **Redis Pub/Sub:** The gateway subscribes to Redis channels `events:user:{userId}`, `events:job:{jobId}`, and `events:account:{accountId}`. The Notification Service and Job Service publish JSON payloads to these channels.
- **Presence Store:** On connection, the gateway writes a hash entry `presence:{userId}:{socketId}` with a TTL of 90 seconds, refreshed by each heartbeat. This allows other services to query whether a user is currently online before deciding to send an email fallback.

### Data It Owns

- **Socket-to-User Mappings:** Ephemeral in-memory maps (`socket.id` → `userId`) plus replicated TTL hashes in Redis.
- **Room Subscription Registries:** Which sockets are joined to `user`, `job`, and `account` rooms. Redis is the source of truth for multi-node consistency; local node caches are secondary.
- **Heartbeat Metadata:** Last `pong` timestamp per socket used for connection hygiene.
- **Offline Backlog (optional):** For critical events only (job failures, account disconnections), the gateway may persist a short backlog in a Redis stream (`backlog:{userId}`) capped at 50 entries with a 24-hour expiry. Non-critical progress updates are dropped if the user is offline.

The gateway does **not** own durable domain data such as user profiles, social account tokens, post content, or Agenda.js job documents; these remain in MongoDB and are accessed via the respective services.

### Failure Modes

| Failure Scenario | Impact | Mitigation |
|---|---|---|
| **Gateway node crash** | All sockets on the node disconnect immediately; clients lose real-time updates until reconnect. | Run stateless nodes behind a load balancer with client-side exponential backoff (1s, 2s, 4s… up to 30s). Redis Pub/Sub ensures surviving nodes continue broadcasting. |
| **Redis outage** | Cross-node message routing fails; each node becomes an isolated silo. | Local in-memory broadcasts continue to serve sockets on the same node. Critical events degrade to email notifications via the Notification Service. Circuit breaker on Redis writes prevents cascading latency. |
| **Auth token expires mid-session** | Client continues receiving updates for a user whose session is no longer valid. | Token re-validation on every `subscribe_*` request. If validation fails, emit `auth.expired` and close the socket with code `1008`. |
| **Thundering herd on reconnect** | Thousands of clients reconnect simultaneously after a deploy, spiking CPU and Redis connections. | Enforce a server-side `max_connections_per_node` limit (e.g., 20,000). Reject with `503` and randomized `Retry-After` headers during the HTTP upgrade phase. |
| **Message buffer overflow** | A slow consumer causes server-side memory to grow as outbound messages queue. | Set `ws` `maxPayload` to 64 KB and per-socket outbound buffer high-water mark to 1 MB. If exceeded, drop non-critical progress events and emit a single `catchup` summary event. |
| **Ghost sockets** | A client disconnects uncleanly (e.g., app kill, network drop), leaving stale entries in Redis presence. | Heartbeat TTL of 90 seconds ensures automatic expiration. A background sweep every 60 seconds reconciles in-memory sockets against Redis presence hashes. |

### Scaling Considerations

- **Horizontal Scaling Metric:** Scale gateway pods based on **concurrent connections** (target ~10,000–15,000 WebSockets per Node.js instance) rather than CPU. Memory is the dominant constraint due to per-socket object overhead.
- **Redis Adapter:** Mandatory for production multi-replica deployments. Use a Redis Cluster or Sentinel-backed Redis Cache to prevent the adapter from becoming a single point of failure.
- **Avoid Sticky Sessions for WebSockets:** Rely on the Redis adapter for broadcast fan-out rather than L7 sticky sessions. This allows the load balancer to use least-connections routing for the initial HTTP upgrade.
- **Binary Payload Prohibition:** The gateway handles JSON control-plane events only. Media payloads (photos/videos) must never traverse WebSockets; clients retrieve processed media URLs from the CDN via HTTPS.
- **Namespace Isolation:** If the platform later exposes public real-time features (e.g., live analytics), separate Socket.io namespaces or distinct gateway deployments should isolate internal job events from public traffic.
- **Observability:** Track `ws.connections.active`, `ws.messages.outbound_per_sec`, `ws.redis_publish_latency_ms`, and `ws.presence.reconciliation_lag`. Alert when connection count approaches 80% of the per-node limit.
